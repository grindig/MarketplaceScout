# i18n Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to execute this plan task-by-task with two-stage review per task.

**Goal:** Make every user-facing string in the bot language-configurable via `json/locales/<lang>.json`. Add English (default) and German. Adding a third language is "drop a new JSON file, register it in `AVAILABLE_LANGUAGES`."

**Architecture:** A single `i18n.py` module that owns:
- `set_language(code: str)` — called once at startup after `load_config()`
- `t(key: str, **kwargs) -> str` — looks up a key in the active locale, falls back to English, raises `KeyError` if missing in *both*
- a `t_ctx(key)` async-context-manager / `flock`-style helper is **not** needed — there's exactly one language active per process and Discord embeds/commands/console all draw from it

JSON files hold flat key→string mappings. `{var}` placeholders use Python `str.format`. **No new dependencies** (no gettext, no babel).

**Tech Stack:** Python stdlib only. `json`, `str.format`. The existing config loader in `main.py` already reads JSON from `json/`.

**Step 3: Write `i18n.py`**

```python
"""Lightweight i18n: load json/locales/<lang>.json at startup, look up strings via t(key, **kwargs).

The active language is set once per process via set_language(code). All lookups
go through t(), which interpolates {vars} with str.format. If a key is missing
in the active language but present in English, English is returned. If it's
missing in both, KeyError is raised — that signals a missing translation, not
a silent fallback to a placeholder.

Adding a new language is two steps:
1. Drop json/locales/<code>.json with the same keys as en.json.
2. Add the code to AVAILABLE_LANGUAGES below.
"""

import json
import os
from pathlib import Path

_LOCALES_DIR = Path(__file__).parent / "json" / "locales"
AVAILABLE_LANGUAGES = ("en", "de")
_FALLBACK = "en"

_LOCALES: dict[str, dict] = {}
_ACTIVE: str = _FALLBACK


def _load(code: str) -> dict:
    path = _LOCALES_DIR / f"{code}.json"
    with open(path, "r", encoding="utf-8") as f:
        return json.load(f)


def _load_all() -> None:
    for code in AVAILABLE_LANGUAGES:
        _LOCALES[code] = _load(code)


def get_language() -> str:
    return _ACTIVE


def set_language(code: str) -> None:
    """Activate a language. Raises ValueError if the code isn't registered."""
    global _ACTIVE
    if code not in AVAILABLE_LANGUAGES:
        raise ValueError(
            f"Unsupported language: {code!r}. "
            f"Available: {', '.join(AVAILABLE_LANGUAGES)}"
        )
    _ACTIVE = code


def t(key: str, **kwargs) -> str:
    """Look up a translation key in the active language, falling back to English.

    If the key is missing in BOTH languages, raises KeyError so a missing
    translation is loud, not silent. {var} placeholders are filled from kwargs.
    """
    template = _LOCALES.get(_ACTIVE, {}).get(key) or _LOCALES[_FALLBACK].get(key)
    if template is None:
        raise KeyError(f"Translation key {key!r} not in {_FALLBACK!r} or {_ACTIVE!r}")
    return template.format(**kwargs) if kwargs else template


_load_all()
```

## Risks / tradeoffs

- **Slash-command sync timing:** Discord caches slash-command metadata. A language change requires a re-sync, which happens in `on_ready`. So a user who changes `language` in `config.json` and restarts will see the new descriptions within ~5 seconds of bot start. Acceptable; no mitigation needed.
- **Backward compatibility:** Existing `config.json` files without a `language` key default to `en`. Users who *want* German have to add `"language": "de"` explicitly. This is a one-time, well-documented migration. The plan does NOT auto-detect from system locale.
- **Format-string escapes:** JSON has no native `{var}` syntax, so a literal `{` or `}` in any translated string must be escaped as `{{` / `}}`. The en.json/de.json must be carefully reviewed for this — e.g. the JSON example in the error message has `{` and `}` in it. If any user-visible string contains a literal brace, escape it.
- **Time formatting:** The midnight-restart log includes a timestamp formatted via `strftime`. The format string `"%d.%m.%Y %H:%M"` is locale-neutral (it doesn't use `%c` or `%x`), so it stays the same. No translation needed.
- **Prices:** `450 €` / `49,99 €` use the German decimal-comma convention. If we ever add `en-US`, `4,950.00 USD` would be the natural form. For now both `en` and `de` use `,`-as-decimal-separator because the marketplace *is* German. Document this in the de.json/en.json headers if you want, but it's not blocking.
- **No pluralization:** `"{n} Inserate"` is hard-coded plural. `"{n} listings"` is hard-coded plural. If we ever add a language with 4 plural forms (e.g. Polish), we'd need to swap `str.format` for a real i18n library. Not a concern for en+de.

## Out of scope (deliberately)

- A `!language` runtime command. The plan keeps language as restart-only (matches the existing `backfill_days` / `stats_channel_id` pattern).
- Per-channel or per-user language overrides. Always global.
- Translating the `Accept-Language` header in `scanner.py` — that's a request header, not bot output.
- Translating embed *content* (the user-supplied listing title, location, etc.) — those are not bot text, they're data.
- Translating log lines in `__init__` / third-party libraries. Out of scope.
- A web UI or config-file schema validator for the `language` key. The runtime error is the validator.
