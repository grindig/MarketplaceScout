# MarketplaceScout v7.0.0 — Docker-first deployment

MarketplaceScout v7.0.0 turns the project from a Python script you run manually into a cleaner, repeatable, Docker Compose based service. The bot can now be deployed, restarted, upgraded, and backed up using the same workflow people expect from long-running self-hosted bots.

This release focuses on **operator experience**: predictable runtime, persistent state, graceful shutdown, and a documented update path.

## Highlights

### Docker Compose is now the recommended production path

The repository now ships with a production-oriented Docker setup:

- `Dockerfile` builds a slim Python 3.13 runtime image.
- `docker-compose.yml` runs MarketplaceScout as a managed service.
- `.dockerignore` keeps secrets, local state, caches, tests, docs, and agent scratch files out of the runtime image.
- The README now documents fresh install, logs, clean shutdown, updates, backups, and the non-Docker fallback.

The standard deployment flow is now:

```bash
git clone https://github.com/grindig/MarketplaceScout.git
cd MarketplaceScout
cp .env.example .env
cp json/config.example.json json/config.json
# edit .env and json/config.json
docker compose up -d --build
```

### Safer long-running container behavior

The Compose service is configured for unattended operation:

- `restart: unless-stopped` brings the bot back after host reboots or process exits.
- `init: true` gives the container a tiny init process for cleaner signal handling and child reaping.
- `stop_signal: SIGINT` and `STOPSIGNAL SIGINT` route Docker stops through the bot's existing graceful shutdown path, so pending seen IDs are flushed instead of being dropped on normal container stops.
- `stop_grace_period: 30s` gives Discord shutdown and state flushing time to finish.
- Docker log rotation is enabled with `10m` files and `3` retained files, avoiding unbounded daemon log growth.

### Persistent state stays on the host

The Compose service bind-mounts the repository's `json/` directory into `/app/json`:

```yaml
volumes:
  - ./json:/app/json
```

That means these files survive image rebuilds and container recreates:

- `json/config.json`
- `json/seen.json`
- `json/prices.json`
- `json/stats_state.json`
- custom edits to `json/keywords.json`

This is intentionally source-built Compose rather than a published image workflow. Updating is explicit and boring:

```bash
git pull
docker compose up -d --build
```

No runtime state is baked into the image.

### Still works without Docker

The existing local Python flow is preserved for development and simple manual runs:

```bash
pip install -r requirements.txt
python main.py
```

Docker is the recommended deployment path, not a forced migration.

## Upgrade guide

### From an older Git checkout

```bash
git pull
cp .env.example .env                 # only if you do not already have .env
cp json/config.example.json json/config.json  # only if you do not already have config.json
# verify/edit .env and json/config.json
docker compose up -d --build
```

### Recommended backup before upgrading

```bash
tar -czf marketplacescout-state-$(date +%F).tar.gz \
  .env \
  json/config.json \
  json/seen.json \
  json/prices.json \
  json/stats_state.json
```

If some state files do not exist yet, that usually just means the bot has not created them yet.

### Check logs after the upgrade

```bash
docker compose logs -f marketplacescout
```

Look for startup, slash-command sync, scan-loop, stats-board, and archive-loop messages.

## Operational notes

### Updating does not create a new Compose file

`docker-compose.yml` is the reusable service recipe. Each version rebuilds or recreates the container from the current source checkout:

```bash
git pull
docker compose up -d --build
```

The Compose file normally stays stable; the image/container changes.

### Why this does not publish a registry image yet

v7.0.0 intentionally starts with source-built Docker Compose because it is the lowest-friction deployment improvement:

- no registry credentials required,
- no GitHub Actions pipeline required,
- no image publishing policy to maintain,
- local config/state stays obvious and inspectable.

A future release can add GHCR publishing so upgrades become:

```bash
docker compose pull
docker compose up -d
```

For now, the documented update command is `docker compose up -d --build`.

### Permissions

The image runs as UID/GID `1000:1000`, matching the common first Linux user. This keeps the bind-mounted `json/` directory writable on most servers. If your host uses a different UID/GID, update the Compose service's `user:` setting or adjust ownership of the `json/` directory.
