1|# MarketplaceScout v7.0.0 — Docker-first deployment
2|
3|MarketplaceScout v7.0.0 turns the project from a Python script you run manually into a cleaner, repeatable, Docker Compose based service. The bot can now be deployed, restarted, upgraded, and backed up using the same workflow people expect from long-running self-hosted bots.
4|
5|This release focuses on **operator experience**: predictable runtime, persistent state, graceful shutdown, and a documented update path.
6|
7|## Highlights
8|
9|### Docker Compose is now the recommended production path
10|
11|The repository now ships with a production-oriented Docker setup:
12|
13|- `Dockerfile` builds a slim Python 3.13 runtime image.
14|- `docker-compose.yml` runs MarketplaceScout as a managed service.
15|- `.dockerignore` keeps secrets, local state, caches, tests, docs, and agent scratch files out of the runtime image.
16|- The README now documents fresh install, logs, clean shutdown, updates, backups, and the non-Docker fallback.
17|
18|The standard deployment flow is now:
19|
20|```bash
21|git clone https://github.com/grindig/MarketplaceScout.git
22|cd MarketplaceScout
23|cp .env.example .env
24|cp cfg/config.example.json cfg/config.json
25|# edit .env and cfg/config.json
26|docker compose up -d --build
27|```
28|
29|### Safer long-running container behavior
30|
31|The Compose service is configured for unattended operation:
32|
33|- `restart: unless-stopped` brings the bot back after host reboots or process exits.
34|- `init: true` gives the container a tiny init process for cleaner signal handling and child reaping.
35|- `stop_signal: SIGINT` and `STOPSIGNAL SIGINT` route Docker stops through the bot's existing graceful shutdown path, so pending seen IDs are flushed instead of being dropped on normal container stops.
36|- `stop_grace_period: 30s` gives Discord shutdown and state flushing time to finish.
37|- Docker log rotation is enabled with `10m` files and `3` retained files, avoiding unbounded daemon log growth.
38|
39|### Persistent state stays on the host
40|
41|The Compose service bind-mounts the repository's `cfg/` directory into `/app/json`:
42|
43|```yaml
44|volumes:
45|  - ./json:/app/json
46|```
47|
48|That means these files survive image rebuilds and container recreates:
49|
50|- `cfg/config.json`
51|- `cfg/seen.json`
52|- `cfg/prices.json`
53|- `cfg/stats_state.json`
54|- custom edits to `cfg/keywords.json`
55|
56|This is intentionally source-built Compose rather than a published image workflow. Updating is explicit and boring:
57|
58|```bash
59|git pull
60|docker compose up -d --build
61|```
62|
63|No runtime state is baked into the image.
64|
65|### Still works without Docker
66|
67|The existing local Python flow is preserved for development and simple manual runs:
68|
69|```bash
70|pip install -r requirements.txt
71|python main.py
72|```
73|
74|Docker is the recommended deployment path, not a forced migration.
75|
76|## Upgrade guide
77|
78|### From an older Git checkout
79|
80|```bash
81|git pull
82|cp .env.example .env                 # only if you do not already have .env
83|cp cfg/config.example.json cfg/config.json  # only if you do not already have config.json
84|# verify/edit .env and cfg/config.json
85|docker compose up -d --build
86|```
87|
88|### Recommended backup before upgrading
89|
90|```bash
91|tar -czf marketplacescout-state-$(date +%F).tar.gz \
92|  .env \
93|  cfg/config.json \
94|  cfg/seen.json \
95|  cfg/prices.json \
96|  cfg/stats_state.json
97|```
98|
99|If some state files do not exist yet, that usually just means the bot has not created them yet.
100|
101|### Check logs after the upgrade
102|
103|```bash
104|docker compose logs -f marketplacescout
105|```
106|
107|Look for startup, slash-command sync, scan-loop, stats-board, and archive-loop messages.

118|## Operational notes
119|
120|### Updating does not create a new Compose file
121|
122|`docker-compose.yml` is the reusable service recipe. Each version rebuilds or recreates the container from the current source checkout:
123|
124|```bash
125|git pull
126|docker compose up -d --build
127|```
128|
129|The Compose file normally stays stable; the image/container changes.
130|
131|### Why this does not publish a registry image yet
132|
133|v7.0.0 intentionally starts with source-built Docker Compose because it is the lowest-friction deployment improvement:
134|
135|- no registry credentials required,
136|- no GitHub Actions pipeline required,
137|- no image publishing policy to maintain,
138|- local config/state stays obvious and inspectable.
139|
140|A future release can add GHCR publishing so upgrades become:
141|
142|```bash
143|docker compose pull
144|docker compose up -d
145|```
146|
147|For now, the documented update command is `docker compose up -d --build`.
148|
149|### Permissions
150|
151|The image runs as UID/GID `1000:1000`, matching the common first Linux user. This keeps the bind-mounted `cfg/` directory writable on most servers. If your host uses a different UID/GID, update the Compose service's `user:` setting or adjust ownership of the `cfg/` directory.
